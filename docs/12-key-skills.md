# 核心技能详解

难度：进阶

本章逐一介绍五个最实用的技能。每个技能都会从"它解决什么问题""它怎么工作""什么时候用"三个角度来说。

## brainstorming（头脑风暴）

**解决的问题**：你有一个模糊的想法，但还没想清楚具体怎么做。比如"我想给应用加个通知系统"，但不知道该用 WebSocket 还是轮询，不知道消息怎么存储，不知道要不要做已读未读。

**工作方式**：

1. 先了解你的项目上下文（读取代码、理解现有架构）
2. 一次只问一个问题，等你的回答后再问下一个
3. 每个设计决策提供 2-3 种方案供你选择
4. 把设计方案分成 200-300 字的小段，逐段给你看，每段确认后再写下一段
5. 最后输出一份完整的设计文档

**核心原则**：
- 选择题优于开放式问题（给你选 A 还是 B，而不是"你想怎么做"）
- 严格遵守 YAGNI（不需要的功能不要加）
- 永远不要替你做技术选型决策，而是给出选项让你选

**输出**：`docs/plans/YYYY-MM-DD-<topic>-design.md`

**什么时候用**：需求模糊、需要先想清楚再动手的任何场景。不限于编码，架构设计、技术选型、流程优化都可以用。

## writing-plans（编写计划）

**解决的问题**：需求已经明确（可能通过了 brainstorming），需要一个详细的实施计划，指导 AI（或人）一步步把代码写出来。

**工作方式**：

1. 假设执行者对代码库零上下文，计划必须包含所有必要信息
2. 把任务拆解为极小的步骤（每步 2-5 分钟可完成）
3. 每个步骤精确到：改哪个文件、改什么内容、写什么代码、运行什么命令、预期什么输出
4. 不写"添加验证"这种模糊描述，而是直接给出要写的完整代码
5. 遵循 DRY（不重复）、YAGNI（不添加不需要的）、频繁提交的原则

**输出**：`docs/plans/YYYY-MM-DD-<feature-name>.md`

**执行交接**：计划写好后，有两种执行方式：
- **子代理驱动**：AI 在当前会话中逐个任务分派执行
- **并行会话**：开一个新会话，把计划丢进去批量执行

**什么时候用**：任何涉及 3 个以上文件、或估计超过 30 分钟的任务，都建议先写计划。

## grill-with-docs（审问式文档化）

**解决的问题**：项目领域有自己的术语和概念，这些概念在代码中可能命名不一致、文档中可能缺失。你需要在动手之前把关键术语和决策统一起来。

**工作方式**：

1. 一次只问一个问题，等待你的回答
2. 能从代码库中自行回答的问题不问你（AI 先查代码，查不到再问人）
3. 把解析出的术语实时写入 `CONTEXT.md`（词汇表）
4. 不可逆的决策记录为 ADR（架构决策记录），保存在 `docs/adr/` 目录

**输出**：
- `CONTEXT.md`：纯术语表，记录项目中每个关键概念的含义。只有术语定义，不写实现细节
- `docs/adr/NNNN-<决策标题>.md`：每个重要决策的完整记录，包括背景、选项、决定、后果

**来源**：社区技能，需要手动安装：

```bash
npx skills add mattpocock/skills --skill=grill-with-docs
```

**什么时候用**：领域知识复杂（金融、医疗、企业应用等）、团队新人多需要统一认知、概念命名混乱需要梳理的场景。

## test-driven-development（TDD）

**解决的问题**：确保代码有足够的测试覆盖，并且在开发过程中不引入回归。

**工作方式**：严格遵循 **Red-Green-Refactor** 循环：

1. **Red**：先写一个会失败的测试，描述你期望的行为
2. **Green**：写最少的代码让测试通过
3. **Refactor**：在不破坏测试的前提下改进代码结构

然后重复这个循环，直到功能完整。

**核心原则**：
- 不为没有失败测试的生产代码编写代码
- 测试描述期望行为，不描述实现
- 目标 80% 以上的覆盖率
- 每个循环尽量小，不要一次写太多测试

**什么时候用**：任何需要长期维护的功能代码。对于原型、实验性代码、一次性脚本，不需要强制 TDD。可以和 writing-plans 配合使用：先在计划中规划好测试文件，再按 TDD 循环逐步实施。

## code-review（代码审查）

**解决的问题**：AI 写完代码（或你写了代码）后，需要检查质量、安全性、一致性和潜在问题。

**工作方式**：

1. 读取变更的文件内容
2. 按检查清单逐项审查：
   - 代码风格是否一致
   - 是否有潜在的 bug 或边界情况
   - 是否有安全风险（SQL 注入、XSS 等）
   - 命名是否清晰
   - 是否有重复代码可以提取
   - 性能是否有明显问题
3. 对每个发现的问题给出具体的修改建议
4. 输出审查报告

**什么时候用**：AI 完成了一个较大的功能后、你写完一段重要的代码后、或合并代码之前。不需要对每一行改动都做 code review，重点关注核心逻辑和复杂变更。
