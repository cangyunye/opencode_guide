# 长时间操作

难度：进阶

## 面临的挑战

让 AI 连续工作半小时甚至更长时间，会遇到几个现实问题：

- **上下文耗尽**：对话越长，AI 需要记住的内容越多。一旦超过模型的上下文窗口，早期的对话内容会被丢弃，AI 会"忘记"之前做了什么
- **中途出错**：某一步操作失败，AI 没有正确恢复，后续步骤全部偏离
- **权限阻塞**：AI 需要你手动批准某个操作，但你不在线，任务就停在那里
- **成本累积**：长时间的对话消耗大量 tokens，费用可能超出预期

了解这些挑战后，我们来看看应对策略。

## 导致任务停止或中断的常见原因

了解中断原因，才能有针对性地预防：

| 原因 | 表现 | 应对 |
|------|------|------|
| 上下文窗口耗尽 | AI 开始输出不相关内容或报错 | 启用自动压缩 |
| 权限被拒绝（deny 规则） | AI 提示某操作被禁止 | 检查权限配置 |
| 权限等待用户确认 | 操作卡住等批准 | 用 --auto 或调整权限 |
| doom_loop 触发 | AI 反复做同一件事后暂停 | 调整描述或拆分任务 |
| 模型 API 限流/超时 | 网络错误提示 | 等待重试或换模型 |
| 子代理执行失败 | 某个步骤报错但 AI 未恢复 | 检查错误信息，/undo 后调整 |
| bash 命令执行错误 | 命令返回非零退出码 | 确保命令正确，或让 AI 先测试 |
| 用户手动中断 | Ctrl+C | 正常行为 |

## 让 Agent 持续工作的策略

### 策略一：用 writing-plans 拆分任务

不要一次性让 AI "帮我实现整个用户系统"。用 writing-plans 先拆成 10 个独立的小步骤，然后让 AI 逐步执行。每步完成后自动进入下一步，中间不需要你干预。

### 策略二：启用自动压缩

在 `opencode.json` 中配置：

```json
{
  "compaction": {
    "auto": true
  }
}
```

当上下文使用达到 95% 时，OpenCode 会自动压缩之前的对话为摘要，AI 继续在新的上下文中工作。你不需要手动执行 `/compact`。

### 策略三：在 AGENTS.md 中写清楚持续工作指令

```markdown
## 工作习惯
- 每完成一个步骤后，立即开始下一个步骤，不要停下来等我确认
- 遇到错误时，先尝试自己修复，修复 2 次仍然失败再问我
- 每完成 3 个步骤后，简要汇报进度
```

这样 AI 知道不需要在每个步骤之间等你。

### 策略四：使用 --auto 模式

```bash
opencode --auto
```

自动批准权限请求，避免任务因为等待你批准而停住。配合 deny 规则保护关键操作：

```json
{
  "permission": {
    "bash": {
      "rm *": "deny",
      "sudo *": "deny",
      "*": "allow"
    }
  }
}
```

### 策略五：合理使用子代理

把大任务拆成独立的子任务，每个子任务分派给一个 General 子代理。子代理在独立的子会话中执行，有自己的上下文空间。即使一个子代理失败，也不会拖垮整个任务。

### 策略六：提高 steps 限制

在代理配置中设置 `steps` 参数（最大迭代次数）：

```json
{
  "agent": {
    "build": {
      "steps": 50
    }
  }
}
```

默认值可能较小，长时间任务需要提高这个上限。

## 会话 Fork：换一种方式试试

如果 AI 在某个方向上走了很久但结果不理想，不要从头开始。用 Fork 从当前会话的某个点分叉出一个新会话，尝试不同的方案。

```
# 在 OpenCode 中
/sessions
# 选择当前会话，按 Fork
```

原始会话保留，新会话从分叉点继续，但你可以给出不同的指引。

## 手动压缩

如果自动压缩没有开启，或者你想在特定时机压缩：

```
/compact
```

AI 会把之前的对话总结成一份摘要。压缩后上下文空间释放，AI 能继续工作。代价是早期的细节可能丢失（摘要不会包含所有细节）。

## 成本控制

长时间任务会消耗大量 tokens。几个省钱技巧：

- 用较便宜的模型执行简单步骤，只在关键决策时切换到高质量模型
- 定期 `/compact`，减少上下文中的冗余内容
- 避免让 AI 读取不必要的大文件，指定精确的文件路径
