# 为什么技能决定了 AI 能走多远

难度：进阶

> 本文档是对[技能系统入门](09-skills-intro.md)中"深度分析"的完整展开。

## 一个常见的困惑

同样安装了 OpenCode，同样选择了 DeepSeek 模型，为什么有人做任务很快就中断了，有人却能持续很久？

答案不在模型本身，而在于**是否通过技能构建了一个自驱工作流**。模型是执行引擎，技能是驾驶程序。没有驾驶程序的引擎，要么原地熄火，要么猛冲一段就抛锚。

## 两种截然不同的体验

### 很快就中断的用户

```
用户：帮我给这个项目添加用户注册功能
AI：好的，我来分析一下...（读了几个文件）
AI：我建议这样做：1. 创建 User 模型 2. 写注册接口 3. 加表单
AI：（直接开始写代码，改了 2 个文件）
AI：好了，注册功能基本完成。还需要你自己补充验证码和邮件确认。
```

看起来"完成了"，但实际上：
- 大量细节被跳过（输入验证、错误处理、数据库迁移）
- AI 主动停下来"交棒"，等待用户反馈
- 用户检查后发现很多问题，不得不反复描述补充需求
- 几轮下来，上下文膨胀，AI 开始遗忘早期内容
- 最终用户觉得"AI 干不了复杂活"

### 能持续很久的用户

```
用户：帮我给这个项目添加用户注册功能
（AI 自动加载 brainstorming 技能）
AI：在开始之前，我想先确认几个细节。第一个问题：
    用户注册需要哪些字段？（用户名、邮箱、手机号，还是其他？）
用户：用户名和邮箱
AI：好的。第二个问题：注册后是否需要邮箱验证？
用户：需要
（...经过 5-6 轮问答，需求完全明确...）
AI：需求明确了，我输出一份设计文档到 docs/plans/...
（AI 自动切换到 writing-plans 技能）
AI：基于设计文档，我制定以下实施计划：
    步骤 1：创建数据库迁移文件
    步骤 2：创建 User 模型和验证逻辑
    ...
    步骤 8：编写端到端测试
    每步完成后自动进入下一步。
（AI 按计划逐步执行，每步写完自动跑测试确认）
```

关键区别：**AI 一直在主动推进，而不是被动等待。**

## 逐层分析中断原因

### 第一层：模型层 -- DeepSeek 的特性

DeepSeek（尤其是 V4 Flash 这类低价模型）有几个特点：

- **上下文保持能力偏弱**：长对话中容易丢失早期信息，导致后续步骤偏离目标
- **自主判断"完成"的阈值低**：写了一点代码就认为"差不多了"，倾向于主动结束
- **多步推理能力不如旗舰模型**：面对需要 10 步以上的任务，可能在第 3-4 步就开始简化或跳步
- **工具调用链较短**：一次回复中连续调用 2-3 个工具后就倾向于停下来

这些不是 bug，而是模型定价带来的自然权衡。Flash 级模型设计目标是"快和便宜"，不是"深和持久"。

### 第二层：行为层 -- 没有"持续"指令

没有技能约束时，AI 的行为模式是：

```
收到任务 → 思考方案 → 执行一步 → 等待反馈
```

这个循环的问题在于"等待反馈"这一步。对于简单任务这没问题，但对于复杂任务，AI 每做一步就停下来问你"这样可以吗？"，效率极低。

### 第三层：结构层 -- 没有"持久记忆"

纯对话模式下，所有的计划、上下文、决策都存在于 AI 的对话上下文中。一旦上下文压缩或溢出，这些信息就丢失了。AI 失去方向感，要么重复做已经做过的事，要么彻底停下来。

## 技能如何逐层解决这些问题

技能并不是让模型变聪明，而是给模型装了一套**结构化的行为框架**，弥补了低价模型在自主性和持续性上的不足。

### writing-plans：解决"持久记忆"问题（第三层）

这是最关键的一个技能。它的核心价值不在于"规划"，而在于**持久化**。

没有 writing-plans 时：
```
任务信息 → 全部存在对话上下文中 → 上下文满了 → 信息丢失 → AI 迷失方向
```

有 writing-plans 时：
```
任务信息 → 写入 docs/plans/xxx.md（磁盘文件）
→ 每步开始前 AI 重新读取计划文件 → 即使上下文被压缩，计划仍然完整
```

计划文件是 AI 的"外部记忆"。无论对话多长、压缩多少次，AI 都能通过重新读取文件恢复完整上下文。这就是为什么用技能的用户能持续很久 -- **他们的 AI 有了一个不会丢失的记事本。**

### brainstorming：解决"返工"导致的上下文浪费（第二层）

没有 brainstorming 时，最常见的失败模式是：

```
AI 按自己的假设做了 5 步 → 用户说"不对，我要的不是这个"
→ /undo → 重新描述 → AI 又按新假设做了 5 步 → 又不对
→ 上下文被大量无效对话塞满 → 可用空间不够 → 中断
```

brainstorming 通过前置的问答环节，把需求完全锁定。后面的每一步都是在确认过的需求上推进，**几乎没有返工**。上下文空间全部用于有效工作，而不是反复试错。

### TDD：内置自纠错机制（行为层 + 结构层）

没有 TDD 时，AI 写完代码直接说"完成了"。如果有 bug，你要到手动测试时才发现，然后回来描述问题，AI 再修。一来一回消耗大量上下文。

有 TDD 时，每一步都是：
```
写测试 → 运行测试（失败是预期的）→ 写代码 → 运行测试（必须通过）
→ 测试没通过？AI 自己修，不问你 → 通过了？自动进入下一步
```

测试就是 AI 的自我检查。测试不通过，AI 不会停下来问你，而是自己修复。**这个过程完全不需要人工干预，任务自然持续下去。**

## 技能链：自适应工作流全貌

当这些技能组合在一起时，形成了一个自适应的工作流：

```
brainstorming（需求锁定）
    ↓ 输出：设计文档（磁盘文件）
writing-plans（步骤拆解 + 持久化）
    ↓ 输出：计划文件（磁盘文件，不丢失）
TDD（每步自检 + 自动修复）
    ↓ 输出：通过的代码和测试
code-review（最终验证）
    ↓ 输出：质量报告
```

这个流程之所以"自适应"，是因为：

1. **需求不确定时**，brainstorming 自动展开问答，不会贸然动手
2. **任务很大时**，writing-plans 自动拆成小步骤，不会一口吃成胖子
3. **某步出错时**，TDD 的测试自动捕获，AI 自动修复，不会带着 bug 往下走
4. **上下文快满时**，计划文件还在磁盘上，AI 重新读取后继续，不会迷失
5. **每步之间不需要人工干预**，AI 自己判断"测试通过了，进入下一步"

## 总结对比

| 维度 | 无技能（容易中断） | 有技能（持续工作） |
|---|---|---|
| 任务理解 | AI 自己猜，经常猜错 | brainstorming 问答确认 |
| 执行计划 | 存在对话中，压缩后丢失 | 写入文件，永久可读 |
| 质量保障 | 做完再说，bug 要人工发现 | TDD 每步自检 |
| 步骤衔接 | 做完一步停下来等用户 | 计划驱动，自动进入下一步 |
| 错误恢复 | 人工发现 -> 描述问题 -> AI 修 | 测试失败 -> AI 自动修复 |
| 上下文效率 | 大量空间被返工和试错占用 | 几乎全部用于有效工作 |
| 中断点 | 每步之间都需要人工确认 | 只在关键决策点需要人工参与 |

**一句话：技能的价值不是让 AI 变强，而是让 AI 的工作方式从"做一步问一步"变成"有计划地持续推进"。对于 DeepSeek 这类性价比模型，这个转变尤为重要 -- 它弥补了模型在自主性和长程推理上的短板。**
