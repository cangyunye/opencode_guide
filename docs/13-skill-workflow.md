# 技能使用顺序

难度：进阶

## 完整开发工作流

在一个理想的大型开发任务中，技能的使用可以编排成一条清晰的流水线。不需要每次都走完全部流程，按需组合。

## 推荐的技能编排顺序

### 第一阶段：想清楚（可选）

**brainstorming** -- 需求模糊时使用

当你只有一个大致想法时，用 brainstorming 把它细化成一份设计文档。产出物是一个清晰的技术方案。

如果你已经有明确的需求文档或 issue 描述，可以跳过这一步。

### 第二阶段：理清术语（可选）

**grill-with-docs** -- 领域复杂时使用

如果项目涉及复杂的业务领域（比如支付系统、库存管理、权限模型），术语和概念可能需要先统一起来。用 grill-with-docs 生成 CONTEXT.md 和 ADR。

如果是纯技术任务（加个搜索功能、重构一个模块），通常不需要。

### 第三阶段：制定计划

**writing-plans** -- 把设计转化为可执行的步骤

有了明确的设计后，用 writing-plans 把它拆解成一个个精确的实施步骤。每一步都精确到文件路径、代码内容、命令和预期输出。

### 第四阶段：编码实施

**test-driven-development** -- 按计划逐步实现

按 writing-plans 的步骤，结合 TDD 的 Red-Green-Refactor 循环，一步步编写代码。每完成一个步骤，运行测试确认通过，然后进入下一步。

### 第五阶段：质量检查

**code-review** -- 实现完成后审查

所有步骤完成后，用 code-review 技能检查代码质量，发现潜在问题。

## 一个完整示例

假设你要给电商系统添加优惠券功能：

```
# 阶段一：想清楚
请使用 brainstorming 技能，帮我设计一个优惠券系统。
功能包括：满减券、折扣券、限时券、用户领取和使用。

（AI 逐个问题确认细节，输出设计文档）
（你可以提出修改意见，完善设计）

# 阶段二：跳过（业务逻辑不复杂）

# 阶段三：制定计划
请使用 writing-plans 技能，基于刚才的设计文档，制定详细的实施计划。

（AI 读取设计文档和现有代码，输出分步实施计划）
（你审查计划，确认无误）

# 阶段四：编码实施
按照计划开始实施，使用 TDD 方式：先写测试，再写代码。

（AI 按计划逐步实现，每步先写测试再写代码）

# 阶段五：质量检查
所有代码写完了，请审查一下质量。

（AI 执行 code-review，输出审查报告）
```

## 不是每次都需要全部技能

**适合用完整流程的场景**：新功能开发（涉及多个模块）、大型重构、从零搭建新模块。

**只需要部分技能的场景**：

| 场景 | 推荐技能 |
|------|---------|
| 修一个小 bug | 不需要技能，直接改 |
| 加一个简单的功能 | writing-plans（可选） |
| 重构一个函数 | TDD |
| 做一次代码审查 | code-review |
| 梳理业务术语 | grill-with-docs |
| 新模块从零搭建 | brainstorming -> writing-plans -> TDD -> code-review |

## 技能之间的衔接

每个技能的产出物就是下一个技能的输入：

```
brainstorming 输出：设计文档
        ↓
writing-plans 读取设计文档，输出：实施计划
        ↓
TDD 按计划逐步实施，输出：代码 + 测试
        ↓
code-review 审查代码，输出：审查报告
```

衔接的关键在于产出物路径的一致性。brainstorming 和 writing-plans 默认都输出到 `docs/plans/` 目录，所以可以无缝衔接。
