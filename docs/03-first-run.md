# 第一次启动

难度：基础

## 进入你的项目

OpenCode 需要在一个具体的项目目录里运行，这样它才能读取和理解你的代码。

```bash
cd /path/to/your/project
```

如果你还没有现成的项目，可以临时创建一个：

```bash
mkdir my-first-opencode-project && cd my-first-opencode-project
git init
```

然后启动 OpenCode：

```bash
opencode
```

你会看到一个终端交互界面，底部有输入框，可以直接打字和 AI 对话。

## /init：让 AI 了解你的项目

第一次在项目里使用 OpenCode 时，运行这个命令：

```
/init
```

OpenCode 会扫描你的项目文件，分析项目结构，然后生成一个叫 `AGENTS.md` 的文件。

这个文件是 AI 的"项目说明书"。它会记录你的项目用了什么技术栈、怎么构建、怎么测试、有哪些特殊的代码规范。以后每次对话，OpenCode 都会先读这个文件，了解你的项目背景。

## AGENTS.md 里写了什么

AGENTS.md 通常包含这些内容：

- **项目概述**：项目是做什么的
- **技术栈**：用了什么语言、框架
- **构建和测试命令**：怎么编译、怎么跑测试、命令的执行顺序
- **项目结构**：目录组织方式，哪层放什么
- **编码规范**：命名风格、注释要求等约定
- **注意事项**：哪些地方容易出错，哪些文件不要动

这个文件不是一次写死不变的。你可以在使用过程中手动修改它，也可以再次运行 `/init` 让 OpenCode 基于最新代码更新它。

## 两个级别的 AGENTS.md

**项目级**：放在项目根目录，只对当前项目生效。建议提交到 Git，让团队成员共享。

**全局级**：放在 `~/.config/opencode/AGENTS.md`，对你所有的项目都生效。适合写个人偏好，比如"用中文回复"、"代码注释用英文"这类通用规则。

如果两个级别的规则有冲突，项目级优先。

## 为什么要把 AGENTS.md 提交到 Git

因为这个文件本质上也是项目文档的一部分。它帮助 OpenCode 理解你的项目，同样也能帮助新加入的同事了解项目。而且提交到 Git 后，团队成员每个人用 OpenCode 时都能获得一致的上下文。

## 推荐预置的全局 AGENTS.md

作为初学者，有一些规则写入全局 AGENTS.md 后会立刻改善 AI 的输出质量。以下是一份适合中国开发者的预置模板，你可以直接使用或按需修改。

### 创建位置

**Linux / macOS**：

```bash
mkdir -p ~/.config/opencode
```

**Windows**：

```powershell
New-Item -ItemType Directory -Path "$env:USERPROFILE\.config\opencode" -Force
```

### 推荐内容

将以下内容保存为 `~/.config/opencode/AGENTS.md`（Linux/macOS）或 `%USERPROFILE%\.config\opencode\AGENTS.md`（Windows）：

```markdown
# 全局规则

## 语言

- 始终使用中文回复
- 代码中的变量名、函数名、类名使用英文（遵循行业惯例）
- 代码注释使用中文

## 代码风格

- 优先使用 TypeScript / Python 3 类型注解
- 遵循项目已有的代码风格，不要自作主张改变
- 缩进使用 2 个空格（不使用 Tab）
- 字符串优先使用单引号（JS/TS），双引号也可以接受

## 工作习惯

- 修改文件前，先简要说明你要改什么、为什么改
- 每次修改后运行相关测试，确认没有引入新的问题
- 如果不确定某个命令是否安全，先问我
- 不要生成 node_modules、__pycache__、.venv、vendor 等依赖目录
- 不要修改 .env 文件，如果需要环境变量配置，修改 .env.example

## 生成规则

- 生成的代码要干净，不留 TODO、FIXME 占位符
- 不要添加用不到的 import 或依赖
- 新增依赖前先确认项目中是否已有类似功能的包
- 生成的文件不要包含 BOM 或多余空白行

## Git 规范

- commit message 使用中文，格式：类型: 简短描述
- 类型包括：feat（新功能）、fix（修复）、refactor（重构）、docs（文档）、style（格式）、test（测试）、chore（杂项）
- 不要在 commit 中提交敏感信息（密钥、密码、token）
```

### 这份模板为什么有用

逐条说明每项规则的用意：

**语言规则**：大多数中国开发者更习惯用中文沟通。明确要求中文回复可以避免 AI 在解释和讨论中频繁切换语言。代码标识符保持英文是因为这是行业通用惯例，混合中文命名反而降低可读性。

**代码风格**：TypeScript 和 Python 3 是目前国内最主流的开发语言。缩进规则看似小事，但 AI 有时会混用 Tab 和空格，导致代码风格不一致。

**工作习惯**："修改前先说明"这条规则非常实用。没有这条规则时，AI 可能直接改文件，你不知道它改了什么。"不要生成依赖目录"这条可以避免 AI 把 `node_modules` 之类的目录内容写进文件（虽然不太常见，但确实会发生）。

**生成规则**：AI 有时会在代码中留 `TODO: implement this` 这种占位符然后告诉你"剩下的部分你自己补"。这几条规则可以约束这种行为。

**Git 规范**：很多中国团队使用中文 commit message，这和主流的开源英文规范不同。提前声明可以让 AI 生成的 commit 符合你团队的习惯。

### 按需调整

上面的模板是通用版本。你可以根据实际情况修改：

- 如果你的团队使用英文 commit message，删掉 Git 规范中的中文要求
- 如果你主要写 Go 或 Java，把代码风格中的 TypeScript/Python 改成对应的语言
- 如果你有常用的框架（比如 React + Next.js），加上框架相关的规范
- 如果你想让 AI 更主动（不频繁问你），在"工作习惯"中加上"不确定的小问题自己判断，只在做重要决策时问我"

## CLAUDE.md 兼容

如果你之前用过 Claude Code，项目中可能有 `CLAUDE.md` 文件。OpenCode 也能识别这个文件作为规则来源，不需要额外配置。优先级顺序是：AGENTS.md > CLAUDE.md > 全局 AGENTS.md。

## 来一个简单试试

现在你已经准备好了。试着对 OpenCode 说：

```
帮我看看这个项目的结构，介绍一下主要目录和文件的作用
```

OpenCode 会读取你的项目文件，给你一份项目结构说明。这就是你和 AI 编程搭档的第一次协作。
